Mục tiêu là vào repo và xử lý nhanh 3 việc: hiểu cấu trúc dự án, tìm và fix bug, thêm feature mới trong giới hạn khoảng 1 giờ.

Ưu tiên chuẩn bị theo hướng tăng tốc độ đọc code, khoanh vùng lỗi nhanh, và giảm thời gian setup khi đề được mở.

1. Chuẩn bị môi trường

  • Cài sẵn các tool sẽ dùng: Git, Docker, Node/Python/Java tương ứng, editor, terminal, test runner.
  • Chuẩn bị sẵn template ghi chú để note nhanh: kiến trúc, bug, giả thuyết, cách reproduce, fix đã thử.
  • Tạo sẵn danh sách lệnh thường dùng: chạy app, chạy test, grep code, format, lint, debug log.
  • Chuẩn bị script hoặc alias để tìm file nhanh, search symbol, xem tree project, diff thay đổi.
  • Kiểm tra trước Claude Code hoặc workflow AI hỗ trợ để dùng được ngay khi repo được phát.

2. Rèn quy trình đọc codebase nhanh

  • Luyện thói quen đọc theo thứ tự: README, config, entrypoint, module chính, test, schema, docs.
  • Chuẩn bị checklist phân tích kiến trúc: luồng request, business flow, matching flow, persistence, queue/cache.
  • Luyện cách vẽ sơ đồ nhanh từ code: input, xử lý, rule matching, output, side effects.
  • Chuẩn bị mẫu mô tả project ngắn gọn: mục đích, thành phần chính, luồng dữ liệu, điểm mở rộng.

3. Chuẩn bị chiến lược tìm bug

  • Luyện quy trình: reproduce bug, khoanh vùng module, đọc log, đọc test, tạo giả thuyết, fix nhỏ, verify.
  • Ôn lại các nhóm bug hay gặp trong matching engine: sort sai, race condition, off-by-one, timeout, precision, rule priority.
  • Chuẩn bị checklist đọc bug: dữ liệu đầu vào nào gây lỗi, có deterministic không, có liên quan concurrency không.
  • Luyện viết test trước hoặc test tái hiện bug thật nhanh để khóa hành vi trước khi sửa.
  • Chuẩn bị mindset ưu tiên bug impact lớn trước: crash, sai kết quả matching, mất dữ liệu, performance nghẽn.

4. Chuẩn bị cho phần thêm feature

  • Ôn cách xác định feature impact: thay đổi model nào, API nào, service nào, test nào, migration nào.
  • Chuẩn bị mẫu câu hỏi tự kiểm tra: feature này ảnh hưởng backward compatibility không, có edge case không.
  • Luyện cách implement tối thiểu nhưng an toàn: ít thay đổi, dễ test, dễ giải thích với ban tổ chức.
  • Chuẩn bị template commit hoặc note thay đổi để trình bày rõ feature đã làm gì và chưa làm gì.

5. Chuẩn bị kỹ năng trình bày

  • Chuẩn bị mẫu mô tả bug fix: nguyên nhân, cách phát hiện, cách sửa, cách verify.
  • Chuẩn bị mẫu mô tả kiến trúc project trong 1-2 phút, tránh lan man.
  • Chuẩn bị cách giải thích trade-off: vì sao chọn fix này, vì sao chưa refactor lớn.
  • Luyện viết summary cuối bài ngắn gọn, có số liệu hoặc ví dụ cụ thể nếu có.

6. Chiến thuật trong 1 giờ thi

  • 5-10 phút đầu: setup, chạy project, chạy test, đọc nhanh cấu trúc tổng thể.
  • 15-20 phút tiếp: khoanh vùng bug, tái hiện lỗi, fix bug nặng trước.
  • 15-20 phút sau: làm feature với phạm vi nhỏ nhưng hoàn chỉnh.
  • 10 phút cuối: chạy lại test, review diff, ghi summary, chuẩn bị phần giải thích.
  • Nếu bị kẹt quá lâu ở một bug, chuyển sang bug khác hoặc feature để tránh mất toàn bộ điểm.
Ưu tiên chuẩn bị nhất: môi trường chạy code thật nhanh, quy trình đọc codebase có hệ thống, kỹ năng viết test tái hiện bug, và template ghi chú để mô tả kiến trúc cùng bug fix rõ ràng khi nộp bài.
